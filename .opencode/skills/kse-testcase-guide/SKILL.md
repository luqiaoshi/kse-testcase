# KubeSphere 新需求测试用例编写指南

## 工作流程

新需求测试用例全流程分为 4 个阶段：

```
获取需求 → 用例规划 → 编写用例(人工审核) → 生成 XMind
```

### 阶段 1：需求分析 & 用例规划

- 阅读需求文档，拆分为独立的功能模块
- 输出：`cases/{模块名}/_需求分析与用例规划.md`
  - 模块划分
  - 核心规则提炼
  - 待测场景清单
  - 待确认项（导航路径、交互细节等）

### 阶段 2：编写用例

- 按模块逐一编写完整测试用例
- 遵循下方"用例规范"中的格式
- 输出：`cases/{模块名}/_模块名-测试用例.md`

### 阶段 3：人工审核

- QA 工程师审核用例完整性、准确性
- 审核通过后进入下一阶段
- 审核不通过则返回阶段 2 修改

### 阶段 4：生成 XMind

- 审核通过后，使用对应脚本生成 XMind 文件
- 方式一（标准 CRUD 模板）：`ai/gen_xmind.py`
- 方式二（标准分组式，步骤-预期 1:1）：`ai/gen_xmind_from_license_md.py`

## 用例规范

### 用例ID命名
```
tc-{P0/P1/P2/P3}-{{用例类型}：{用例名称}
```

**用例类型**：功能测试、性能测试、安全性测试、高可用测试、接口测试

**优先级说明**：
| 优先级 | 说明 |
|--------|------|
| P0 | 核心功能冒烟用例，必须通过 |
| P1 | 重要功能用例，正常流程覆盖 |
| P2 | 次要功能用例，边界情况覆盖 |
| P3 | 详细校验用例，字段/状态验证 |

### 用例结构

**标准分组式** — 步骤和结果分成两个独立章节，且必须一一对应。

```markdown
## tc-P1-功能测试：{用例名称}

### 前置条件 (pc)
- {pc描述}

### 测试步骤
1. 进入【导航路径】
2. 具体操作...
3. 具体操作...

### 预期结果
1. 步骤1的预期结果
2. 步骤2的预期结果
3. 步骤3的预期结果
   1. 检查点1
   2. 检查点2
```

**步骤与预期结果对应规则**：
1. **一一对应**：每个测试步骤必须在「预期结果」中有且仅有一个对应的顶层 `N. ` 条目。步骤序号 `N.` 与预期结果顶层条目的顺序必须完全一致。
2. **多检查点缩进**：如果一个步骤需要多个检查点，请将这些检查点作为缩进子项放在该步骤对应的顶层条目下。缩进使用三个空格，子项前**不加 `- `**前缀，使用 `1. 2. 3.` 编号。
3. **避免冗余**：预期结果中不要重复测试步骤的动作描述，直接写验证点/断言。
4. **禁止混排**：不允许一个步骤对应多个顶层 `N. ` 条目，也不允许多个步骤合并为一个顶层 `N. ` 条目。

**正确示例（1:1 对应）**：

```markdown
### 测试步骤
1. 进入【导航路径】
2. 点击「创建」按钮
3. 输入必填项并点击「保存」

### 预期结果
1. 页面正确加载，显示列表数据
2. 弹出创建弹窗，字段校验规则正确
3. 创建成功，列表同步刷新并展示新增数据
   1. 页面顶部提示「创建成功」
   2. 列表首行显示新增数据
   3. 数据状态为「正常」
```

**错误示例（不要这样写）**：

```markdown
### 测试步骤
1. 进入【导航路径】
2. 点击「创建」按钮
3. 输入必填项并点击「保存」

### 预期结果
- 页面正确加载
- 弹出创建弹窗
- 字段校验规则正确
- 创建成功
- 列表刷新
- 显示新增数据
```

## 用例模板

**核心原则**：测试步骤和预期结果**一一对应**，每一步操作对应一个顶层预期结果条目。
- **操作步骤**：带操作动词（进入、点击、输入、选择、切换、弹出等）
- **预期结果**：描述性语句（显示xxx、成功xxx、列表xxx、不可保存、字段值正确等）

### List 用例

```markdown
## tc-P1-功能测试：列表数据显示正确

### 前置条件 (pc)
- 存在{模块名}数据

### 测试步骤
1. 进入【导航路径】
2. 进入列表页
3. 查看列表数据展示

### 预期结果
1. 页面正确加载，进入列表页
2. 列表数据正常展示
3. 各字段显示正确
```

### Add 用例

```markdown
## tc-P1-功能测试：创建{模块名}，成功

### 前置条件 (pc)
- {pc描述}

### 测试步骤
1. 进入【导航路径】，点击：创建
2. 弹出创建页
3. 输入正确数据，点击：创建

### 预期结果
1. 弹出创建弹窗
2. 创建页字段展示正确
3. 创建成功，列表显示新增数据
```

### Edit 用例

```markdown
## tc-P1-功能测试：编辑{模块名}，成功

### 前置条件 (pc)
- 存在{模块名}数据

### 测试步骤
1. 进入【导航路径】，操作列：编辑
2. 弹出编辑框
3. 修改正确信息，点击：保存

### 预期结果
1. 列表页正常展示
2. 弹出编辑弹窗，字段回显正确
3. 编辑成功，列表同步更新数据
```

## XMind 文件生成模式

> **重要：仅在用户明确下发"生成 XMind"或"生成 XMind 用例"命令时才生成 XMind 文件。编写/补充/输出测试用例时，默认只输出 markdown 文件，不主动生成 XMind。**

### 触发方式
- **仅当用户明确说"生成 XMind"或"生成 XMind 用例"时，才生成 XMind 文件。**
- 如果用户仅要求"编写测试用例"、"补充测试用例"或"输出测试用例"，则只输出 markdown 测试用例文件，不自动生成 XMind。
- 根据用例组织形式选择对应的生成脚本。

### 方式一：标准分组式（模板脚本）

使用 `ai/gen_xmind.py`，通过命令行参数从模板生成标准 CRUD 用例：

```bash
python3 ai/gen_xmind.py \
  --feature "弹性伸缩" --module "容器水平伸缩" --page "HPA" \
  --component "ks-core" --version "4.2.1" \
  --output /path/to/output.xmind
```

也支持 JSON 自定义输入：

```bash
python3 ai/gen_xmind.py --json /path/to/cases.json --output output.xmind
```

### 方式二：标准分组式（从自定义 md 生成 — 步骤-预期 1:1 对应）

适用于按模块组织的复杂需求（如 License 优化），用例采用**标准分组式**排版，但要求步骤和预期结果一一对应。

使用 `ai/gen_xmind_from_license_md.py`，直接解析单文件中的所有用例：

```bash
python3 ai/gen_xmind_from_license_md.py cases/license/_License优化-测试用例.md \
  -o cases/license/License优化-testcase.xmind \
  --feature "License优化" --version "4.2.1"
```

**markdown 格式要求**：

```markdown
## 模块 N：模块名称

---

### tc-P1-功能测试：用例名称

#### 前置条件 (pc)
- 前置条件描述

#### 测试步骤
1. 步骤1
2. 步骤2
3. 步骤3

#### 预期结果
1. 步骤1的预期结果
2. 步骤2的预期结果
3. 步骤3的预期结果
   1. 检查点1
   2. 检查点2     ← 同一预期中的多个检查点，缩进排列
   3. 检查点3
```

**关键规则**：
1. 用例按 `## 模块` 或 `## 专项` 分节
2. `#### 测试步骤` 中用 `N. ` 编号
3. `#### 预期结果` 中用 `N. ` 开头分组，每行对应一个步骤的预期
4. 同一预期中的多个检查点用缩进排列（不带 `- ` 前缀），XMind 中合并在一个叶子节点，换行分隔

**XMind 树结构**：

```
{功能名} - {版本}
├── 模块 N：模块名称
│   └── tc-P1-功能测试：用例名称
│       ├── pc：前置条件
│       ├── 1. 步骤1
│       │   └── 预期结果
│       ├── 2. 步骤2
│       │   └── 预期结果
│       └── 3. 步骤3
│           └── 1. 检查点1\n2. 检查点2\n3. 检查点3
├── 国际化（i18n）专项
│   └── ...
└── API 接口校验专项
    └── ...
```

### 方式三：按模块 CRUD 归类生成合并 XMind

适用于一个目录下存在多个模块的 markdown 测试用例文件，需要合并为一个完整 XMind 文件，并按模块 + CRUD 操作类型归类，方便按模块分派给不同 QA 同事执行。

使用 `ai/gen_xmind_from_dir.py`：

```bash
python3 ai/gen_xmind_from_dir.py cases/{模块目录} \
  -o cases/{模块目录}/{模块目录}-testcase.xmind \
  --feature "{功能名}" --version "4.x"
```

**示例**：

```bash
python3 ai/gen_xmind_from_dir.py cases/byd \
  -o cases/byd/BYD测试用例.xmind \
  --feature "BYD测试用例" --version "4.x"
```

**markdown 源文件格式要求**：

与方式二相同，采用标准分组式排版：

```markdown
## 模块 N：模块名称

### tc-P1-功能测试：用例名称

#### 前置条件 (pc)
- 前置条件描述

#### 测试步骤
1. 步骤1
2. 步骤2

#### 预期结果
1. 步骤1的预期结果
2. 步骤2的预期结果
```

**自动归类规则**（按用例标题关键词，优先级从高到低）：

| 类别 | 命名 | 关键词 |
|------|------|--------|
| 列表 | `{模块名} 列表` | 列表、搜索、分页、排序、刷新、空状态、查询 |
| 新增 | `新增 {模块名}` | 创建、新增、邀请、注册 |
| 编辑 | `编辑 {模块名}` | 编辑、修改 |
| 删除 | `删除 {模块名}` | 删除、移除 |
| 详情 | `{模块名} 详情` | 详情 |
| 其它 | `{模块名} 其它` | 权限、国际化、边界、异常、稳定性、兼容性等 |

> 每个模块只生成实际有数据的类别，无某类用例时不生成空节点。

**XMind 树结构**：

```
{功能名} - {版本}
├── {模块 A}
│   ├── {模块 A} 列表
│   │   ├── tc-P1-功能测试：列表数据显示正确
│   │   │   ├── pc：前置条件
│   │   │   ├── 1. 步骤1
│   │   │   │   └── 预期结果
│   │   │   └── 2. 步骤2
│   │   │       └── 预期结果
│   │   └── ...
│   ├── 新增 {模块 A}
│   ├── 编辑 {模块 A}
│   ├── 删除 {模块 A}
│   ├── {模块 A} 详情
│   └── {模块 A} 其它
├── {模块 B}
│   ├── {模块 B} 列表
│   └── {模块 B} 其它
└── ...
```

**使用建议**：

- 一个目录对应一个完整功能或需求域（如 `cases/byd`）
- 目录下每个 markdown 文件对应一个独立模块（如 `_Region管理-测试用例.md`）
- 模块名从文件 H1 标题自动提取（去掉末尾的 ` - 测试用例` 或 ` 测试用例`）
- 生成后按模块分派给 QA 同事，每个人负责该模块下的列表/新增/编辑/删除/详情/其它用例

### 命名规则
- 中文需求：使用中文短横线风格（如：`GPU管理-算力节点`）
- 英文需求：使用短 slug（如：`gpu-management-computing-node`）
- XMind 文件默认与 md 文件同名（`.xmind` 后缀）

## 注意事项

1. **前置条件**：每个用例需明确 pc（precondition），无前置条件时省略
2. **排版方式**：统一使用**标准分组式**。步骤集中写在「测试步骤」，预期集中写在「预期结果」，各自独立一个章节。必须严格遵循「步骤与预期结果一一对应」规则。
3. **语义区分**：通过内容语义区分步骤和结果
   - **操作步骤**：带操作动词（进入、点击、输入、选择、切换、弹出等）
   - **预期结果**：描述性语句（显示xxx、成功xxx、列表xxx、不可保存、字段值正确等）
4. **权限覆盖**：集群侧和企业空间侧、不同角色（platform-admin/cluster-admin/ws-admin/项目角色）
5. **国际化**：中英文两种语言验证
6. **状态覆盖**：正常/异常/创建中/运行中/告警/离线 等状态
7. **关联操作**：List 与 Add/Edit/Del 联动验证
8. **字段校验**：名称、别名、描述、格式、长度、特殊字符、XSS 漏洞
9. **语言匹配**：生成 XMind 时，标题语言与用户需求语言保持一致
10. **YAML 操作**：编辑/查看 YAML，切换至 YAML 创建
11. **XMind 生成时机**：编写/补充测试用例时**不主动生成 XMind**；仅在用户明确说"生成 XMind"或"生成 XMind 用例"时才生成
