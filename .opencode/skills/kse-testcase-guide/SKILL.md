# KubeSphere 新需求测试用例编写指南

## 工作流程

新需求测试用例全流程分为 4 个阶段：

```
获取需求 → 用例规划 → 编写用例(人工审核) → 生成 XMind
```

### 阶段 1：需求分析 & 用例规划

- 阅读需求文档，拆分为独立的功能模块
- 输出：`cases/{模块名}/_需求分析与用例规划.md`
  - 模块划分
  - 核心规则提炼
  - 待测场景清单
  - 待确认项（导航路径、交互细节等）

### 阶段 2：编写用例

- 按模块逐一编写完整测试用例
- 遵循下方"用例规范"中的格式
- 输出：`cases/{模块名}/_模块名-测试用例.md`

### 阶段 3：人工审核

- QA 工程师审核用例完整性、准确性
- 审核通过后进入下一阶段
- 审核不通过则返回阶段 2 修改

### 阶段 4：生成 XMind

- 审核通过后，使用对应脚本生成 XMind 文件
- 方式一（标准 CRUD 模板）：`ai/gen_xmind.py`
- 方式二（步骤内嵌预期式）：`ai/gen_xmind_md.py`
- 方式三（标准分组式，步骤-预期 1:1）：`ai/gen_xmind_from_license_md.py`

## 用例规范

### 用例ID命名
```
tc-{P0/P1/P2/P3}-{{用例类型}：{用例名称}
```

**用例类型**：功能测试、性能测试、安全性测试、高可用测试、接口测试

**优先级说明**：
| 优先级 | 说明 |
|--------|------|
| P0 | 核心功能冒烟用例，必须通过 |
| P1 | 重要功能用例，正常流程覆盖 |
| P2 | 次要功能用例，边界情况覆盖 |
| P3 | 详细校验用例，字段/状态验证 |

### 用例结构

支持两种排版方式：

**方式一：标准分组式** — 步骤和结果分成两个独立章节

```markdown
## tc-P1-功能测试：{用例名称}

### 前置条件 (pc)
- {pc描述}

### 测试步骤
1. 进入【导航路径】
2. 具体操作...
3. 具体操作...

### 预期结果
- 步骤1的预期结果
- 步骤2的预期结果
- 步骤3的预期结果
  1. 检查点1
  2. 检查点2
```

**步骤与预期结果对应规则（标准分组式）**：
1. **一一对应**：每个测试步骤必须在「预期结果」中有且仅有一个对应的顶层 `- ` 条目。步骤序号 `N.` 与预期结果顶层条目的顺序必须完全一致。
2. **多检查点缩进**：如果一个步骤需要多个检查点，请将这些检查点作为缩进子项放在该步骤对应的顶层条目下。缩进使用两个空格，子项前**不加 `- `**前缀，使用 `1. 2. 3.` 编号。
3. **避免冗余**：预期结果中不要重复测试步骤的动作描述，直接写验证点/断言。
4. **禁止混排**：不允许一个步骤对应多个顶层 `- ` 条目，也不允许多个步骤合并为一个顶层 `- ` 条目。

**正确示例（标准分组式，1:1 对应）**：

```markdown
### 测试步骤
1. 进入【导航路径】
2. 点击「创建」按钮
3. 输入必填项并点击「保存」

### 预期结果
- 页面正确加载，显示列表数据
- 弹出创建弹窗，字段校验规则正确
- 创建成功，列表同步刷新并展示新增数据
  1. 页面顶部提示「创建成功」
  2. 列表首行显示新增数据
  3. 数据状态为「正常」
```

**错误示例（不要这样写）**：

```markdown
### 测试步骤
1. 进入【导航路径】
2. 点击「创建」按钮
3. 输入必填项并点击「保存」

### 预期结果
- 页面正确加载
- 弹出创建弹窗
- 字段校验规则正确
- 创建成功
- 列表刷新
- 显示新增数据
```

**方式二：步骤内嵌预期式** — 步骤和预期用 `→` 分隔，无需 `预期：` 前缀（导入用例管理平台时自动回填到"预期结果"字段）

```markdown
## tc-P1-功能测试：{用例名称}

### 前置条件 (pc)
- {pc描述}

### 测试步骤 & 预期结果
1. 进入【导航路径】 → 页面显示正确
2. 具体操作... → 操作成功
3. 具体操作... → 验证点...
```

## 用例模板

**核心原则**：测试步骤和预期结果**交替排列**，每一步操作对应一步结果，通过语义区分：
- **操作步骤**：带操作动词（进入、点击、输入、选择、切换、弹出等）
- **预期结果**：描述性语句（显示xxx、成功xxx、列表xxx、不可保存等）

支持两种排版方式，以下每种 CRUD 操作均提供**标准分组式**和**步骤内嵌预期式**两种模板。

### List 用例

**标准分组式：**
```markdown
## tc-P1-功能测试：列表数据显示正确

### 前置条件 (pc)
- 存在{模块名}数据

### 测试步骤
1. 进入【导航路径】
2. 进入列表页
3. 查看列表数据展示

### 预期结果
- 列表数据正常展示
- 各字段显示正确
```

**步骤内嵌预期式：**
```markdown
## tc-P1-功能测试：列表数据显示正确

### 前置条件 (pc)
- 存在{模块名}数据

### 测试步骤 & 预期结果
1. 进入【导航路径】 → 进入列表页
2. 查看列表数据展示 → 列表数据正常展示，各字段显示正确
```

## XMind 文件生成模式

### 触发方式
- **仅当用户明确说"生成 XMind"或"生成 XMind 用例"时，才生成 XMind 文件。**
- 如果用户仅要求"编写测试用例"、"补充测试用例"或"输出测试用例"，则只输出 markdown 测试用例文件，不自动生成 XMind。
- 根据用例编写方式选择对应的生成脚本。

### 方式一：标准分组式（模板脚本）

使用 `ai/gen_xmind.py`，通过命令行参数从模板生成标准 CRUD 用例：

```bash
python3 ai/gen_xmind.py \
  --feature "弹性伸缩" --module "容器水平伸缩" --page "HPA" \
  --component "ks-core" --version "4.2.1" \
  --output /path/to/output.xmind
```

也支持 JSON 自定义输入：

```bash
python3 ai/gen_xmind.py --json /path/to/cases.json --output output.xmind
```

### 方式二：步骤内嵌预期式（从 md 目录生成）

使用 `ai/gen_xmind_md.py`，扫描模块目录下所有 md 文件，按文件名前缀自动归类，生成一个完整的模块 XMind。

**目录结构约定**：

```
cases/{模块名}/
├── 列表-{用例描述}.md     → 归类到「1. 列表查询」
├── 创建-{用例描述}.md     → 归类到「2. 新增数据」
├── 编辑-{用例描述}.md     → 归类到「3. 编辑数据」
├── 详情-{用例描述}.md     → 归类到「4. 详情查看」
└── 删除-{用例描述}.md     → 归类到「5. 删除数据」
```

**XMind 树结构**（步骤描述为父节点，预期结果为子节点）：

```

├── 1. 列表查询
│   └── tc-P1-功能测试：{用例名称}

├── 2. 新增数据
│   └── tc-P1-功能测试：{用例名称}

│       │   └── {预期结果}
```

**使用命令**：

```bash
# 从目录生成（推荐）
python3 ai/gen_xmind_md.py cases/{模块名} -o cases/{模块名}/{模块名}.xmind

# 从单个文件生成
python3 ai/gen_xmind_md.py cases/{模块名}/{文件名}.md -o cases/{模块名}/{文件名}.xmind
```

可选参数：
- `--feature`：功能名称（默认：目录名）
- `--version`：版本号（默认：4.x）

**markdown 源文件格式要求**：

```markdown
## tc-P1-功能测试：{用例名称}

### 前置条件 (pc)
- {前置条件}

### 测试步骤 & 预期结果
1. {步骤描述} → {预期结果}
2. {步骤描述} → {预期结果}
```

脚本会自动：
1. 扫描目录下所有 `.md` 文件，按文件名前缀（列表/创建/编辑/详情/删除）归类到对应 CRUD 类别
2. 从 `## tc-` 提取用例标题、从 `pc` 提取前置条件、从 `→` 解析步骤和预期
3. 生成一个完整的 XMind：模块为根 → CRUD 类别 → 测试用例 → 前置条件 + 步骤（父节点）+ 预期（子节点）

### 方式三：标准分组式（从自定义 md 生成 — 步骤-预期 1:1 对应）

适用于按模块组织的复杂需求（如 License 优化），用例采用**标准分组式**排版，但要求步骤和预期结果一一对应。

使用 `ai/gen_xmind_from_license_md.py`，直接解析单文件中的所有用例：

```bash
python3 ai/gen_xmind_from_license_md.py cases/license/_License优化-测试用例.md \
  -o cases/license/License优化-testcase.xmind \
  --feature "License优化" --version "4.2.1"
```

**markdown 格式要求**：

```markdown
## 模块 N：模块名称

---

### tc-P1-功能测试：用例名称

#### 前置条件 (pc)
- 前置条件描述

#### 测试步骤
1. 步骤1
2. 步骤2
3. 步骤3

#### 预期结果
- 步骤1的预期结果
- 步骤2的预期结果
- 步骤3的预期结果
  1. 检查点1
  2. 检查点2     ← 同一预期中的多个检查点，缩进排列
  3. 检查点3
```

**关键规则**：
1. 用例按 `## 模块` 或 `## 专项` 分节
2. `#### 测试步骤` 中用 `N. ` 编号
3. `#### 预期结果` 中用 `- ` 开头分组，每行对应一个步骤的预期
4. 同一预期中的多个检查点用缩进排列（不带 `- ` 前缀），XMind 中合并在一个叶子节点，换行分隔

**XMind 树结构**：

```
{功能名} - {版本}
├── 模块 N：模块名称
│   └── tc-P1-功能测试：用例名称
│       ├── pc：前置条件
│       ├── 1. 步骤1
│       │   └── 预期结果
│       ├── 2. 步骤2
│       │   └── 预期结果
│       └── 3. 步骤3
│           └── 1. 检查点1\n2. 检查点2\n3. 检查点3
├── 国际化（i18n）专项
│   └── ...
└── API 接口校验专项
    └── ...
```

### 命名规则
- 中文需求：使用中文短横线风格（如：`GPU管理-算力节点`）
- 英文需求：使用短 slug（如：`gpu-management-computing-node`）
- XMind 文件默认与 md 文件同名（`.xmind` 后缀）

## 注意事项

1. **前置条件**：每个用例需明确 pc（precondition），无前置条件时省略
2. **排版方式**：支持两种排版方式
   - **标准分组式**：步骤集中写在「测试步骤」，预期集中写在「预期结果」，各自独立一个章节。必须严格遵循「步骤与预期结果一一对应」规则。
   - **步骤内嵌预期式**：步骤标题为「测试步骤 & 预期结果」，每行用 `→` 将步骤和预期写在同一行，无需加"预期："前缀（导入平台时自动回填到"预期结果"字段）
3. **语义区分**：通过内容语义区分步骤和结果
   - **操作步骤**：带操作动词（进入、点击、输入、选择、切换、弹出等）
   - **预期结果**：描述性语句（显示xxx、成功xxx、列表xxx、不可保存、字段值正确等）
4. **权限覆盖**：集群侧和企业空间侧、不同角色（platform-admin/cluster-admin/ws-admin/项目角色）
5. **国际化**：中英文两种语言验证
6. **状态覆盖**：正常/异常/创建中/运行中/告警/离线 等状态
7. **关联操作**：List 与 Add/Edit/Del 联动验证
8. **字段校验**：名称、别名、描述、格式、长度、特殊字符、XSS 漏洞
9. **语言匹配**：生成 XMind 时，标题语言与用户需求语言保持一致
10. **YAML 操作**：编辑/查看 YAML，切换至 YAML 创建
11. **XMind 生成时机**：编写/补充测试用例时**不主动生成 XMind**；仅在用户明确说"生成 XMind"或"生成 XMind 用例"时才生成
